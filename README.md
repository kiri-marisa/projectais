# FitManager — CRM-система для спортивного клуба

**FitManager** — учебный проект CRM-системы для спортивного клуба, разработанный в рамках дисциплины «Администрирование информационных систем».

Проект состоит из backend-части на **FastAPI** и frontend-части на **React/Vite**. Backend отвечает за REST API, работу с MariaDB, CRUD-операции, тестирование и нагрузочные сценарии. Frontend находится в каталоге `site/my-portfolio` и используется как клиентская часть проекта.

---

## 🚀 Технологии

### Backend

- **FastAPI** — веб-фреймворк для реализации REST API.
- **SQLAlchemy** — ORM для работы с базой данных.
- **MariaDB / MySQL** — реляционная база данных.
- **PyMySQL / mysql-connector-python** — драйвер подключения Python к MariaDB/MySQL.
- **Pydantic** — описание и валидация схем данных.
- **Uvicorn** — ASGI-сервер для запуска FastAPI-приложения.
- **unittest** — тестирование CRUD-операций.
- **Locust** — нагрузочное тестирование REST API.

### Frontend

- **React** — библиотека для создания пользовательского интерфейса.
- **Vite** — инструмент сборки frontend-приложения.
- **JavaScript / JSX** — логика клиентской части.
- **HTML5 / CSS3** — структура и стили интерфейса.
- **ESLint** — проверка качества JavaScript-кода.

### Среда разработки и эксплуатации

- **Python 3.11+**
- **Node.js / npm**
- **Debian Linux** на виртуальной машине
- **Windows 10/11** для работы через PowerShell и запуска Locust
- **DBeaver** для просмотра структуры базы данных
- **VirtualBox** для запуска виртуальной машины

## 🎯 Функциональность

### Пользователи и роли

В системе предусмотрены роли:

- **admin** — администратор;
- **trainer** — тренер;
- **client** — клиент.

Основные возможности backend:

- создание пользователей;
- получение списка пользователей;
- получение пользователя по ID;
- изменение email пользователя;
- удаление пользователя;
- просмотр списка ролей.

### Тренировки

Система позволяет:

- получать список тренировок;
- получать тренировку по ID;
- создавать тренировку;
- привязывать тренировку к тренеру;
- привязывать тренировку к помещению;
- задавать дату, время, описание и максимальное количество участников;
- изменять статус тренировки.

### Помещения

Система поддерживает работу с помещениями спортивного клуба:

- создание помещения;
- получение списка помещений;
- указание вместимости помещения.

### Записи на тренировки

Реализована логика записи клиента на тренировку:

- запись клиента на выбранную тренировку;
- проверка существования пользователя;
- проверка существования тренировки;
- запрет повторной записи на одну и ту же тренировку;
- запрет записи на отменённую тренировку;
- проверка свободных мест;
- отмена записи.

### Абонементы

Система поддерживает типы абонементов:

- название;
- стоимость;
- срок действия;
- количество тренировок;
- описание.

### Активные абонементы

Для клиентов можно создавать активные абонементы с указанием:

- пользователя;
- типа абонемента;
- даты начала;
- даты окончания;
- оставшегося количества тренировок;
- статуса.

### Заявки на абонементы

Клиент может оставить заявку на абонемент, а администратор может изменить её статус.

Поддерживаемые статусы:

- `pending` — в ожидании;
- `approved` — одобрена;
- `rejected` — отклонена.

## 🧱 Модель базы данных

В проекте используются следующие таблицы:

- `roles` — роли пользователей;
- `users` — пользователи системы;
- `rooms` — помещения спортивного клуба;
- `trainings` — тренировки;
- `bookings` — записи клиентов на тренировки;
- `subscriptions` — типы абонементов;
- `memberships` — активные абонементы пользователей;
- `sub_requests` — заявки на абонементы.

Связи между таблицами:

```text
roles 1 ─── N users

users (trainer) 1 ─── N trainings
rooms 1 ─── N trainings

users (client) 1 ─── N bookings
trainings 1 ─── N bookings

subscriptions 1 ─── N memberships
users 1 ─── N memberships

subscriptions 1 ─── N sub_requests
users 1 ─── N sub_requests
```

Войдите в MariaDB:

```bash

```


## 📊 Нагрузочное тестирование

Для нагрузочного тестирования используется Locust.

### Подготовка

FastAPI запускается на виртуальной машине:

```bash
python -m uvicorn main:app --host 0.0.0.0 --port 8000
```

На Windows открывается SSH-туннель к MariaDB:

```powershell
ssh -L 3307:127.0.0.1:3306 root@192.168.56.104 -p 22
```

Окно с туннелем необходимо оставить открытым.

### Этап 1 — GET + PUT

Цель: проверить работу системы при операциях чтения и обновления данных при загрузке CPU не выше 60%.

```powershell
locust -f locustfile.py --headless --host=http://192.168.56.104:8000 --tags get_put -u 80 -r 10 --run-time 3m
```

### Этап 2 — GET + PUT + POST

Цель: добавить операции создания записей и заявок, затем сравнить результаты с первым этапом.

```powershell
locust -f locustfile.py --headless --host=http://192.168.56.104:8000 --tags with_post -u 60 -r 8 --run-time 3m
```

### Этап 3 — стресс-тест

Цель: подобрать максимальное количество пользователей при загрузке CPU 90–100% и доле отказов не более 1%.

```powershell
locust -f locustfile.py --headless --host=http://192.168.56.104:8000 --tags with_post -u 120 -r 15 --run-time 5m
```

Для мониторинга нагрузки на виртуальной машине используется:

```bash
htop
```

## 🔒 Ограничения и особенности

В текущей учебной версии проекта реализованы роли пользователей на уровне базы данных. Полноценная JWT-аутентификация и строгая проверка доступа по ролям в API не реализованы.

Реализованные проверки бизнес-логики:

- проверка существования пользователя перед созданием записи;
- проверка существования тренировки;
- запрет повторной записи клиента на одну тренировку;
- запрет записи на отменённую тренировку;
- проверка количества свободных мест;
- уникальность email и номера телефона на уровне базы данных.
